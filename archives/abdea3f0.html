前言

这篇文章讲了三种比较简单且常用的三种设计模式，分别是创建型里的单例模式、行为型里的策略模式、结构型里的代理模式。基本的设计模式有23种，设计模式系列会坚持更新下去!!!（23种啊！天呐！😭先别哭！相信学完以后就会💪，暂且这么安慰一下自己😂）。此处强调一下！学设计模式需要一些必备的基础，如Java面向对象（继承、封装、多态……）、锁的概念以及Java的synchronized用法、数据结构及排序算法、Java多线程用法……在阅读中有什么疑问可在下方评论区评论！

设计模式的概念与意义

  1. 软件设计模式的概念

    软件设计模式（Software Design Pattern），是一套被反复使用、多数人知晓的、经过分类编目的、代码设计经验的总结。它描述了在软件设计过程中的一些不断重复发生的问题，以及该问题的解决方案。也就是说，它是解决特定问题的一系列套路，是前辈们的代码设计经验的总结，具有一定的普遍性，可以反复使用。其目的是为了提高代码的可重用性、代码的可读性和代码的可靠性。

  1. 学习设计模式的意义

    设计模式的本质是面向对象设计原则的实际运用，是对类的封装性、继承性和多态性以及类的关联关系和组合关系的充分理解。

    好处：

    • 提高自身的思维能力、编程能力和设计能力。
    • 使程序设计更加标准化、工程化，提高开发效率。
    • 使设计的代码可重用性高、可维护性强。

      设计模式的分类

单例模式

单例模式（Singleton）的定义：指一个类只有一个实例，且该类能自行创建这个实例的一种模式，单例模式属于创建型模式。

使用场景：

单例模式只允许创建一个实例，当你想控制实例数目、节省系统资源的时候是比较适用的。如多个模块使用同一个数据源连接对象等等。如：

  • 需要频繁实例化然后销毁的对象。
  • 创建对象时耗时过多或者耗资源过多，但又经常用到的对象。
  • 有状态的工具类对象。
  • 频繁访问数据库或文件的对象。

应用案例：

  • 网站的计数器，一般也是采用单例模式实现，否则难以同步。

  • 数据库连接池的设计一般也是采用单例模式，因为数据库连接是一种数据库资源。数据库软件系统中使用数据库连接池，主要是节省打开或者关闭数据库连接所引起的效率损耗。

  • Web应用的配置对象的读取，一般也应用单例模式，这个是由于配置文件是共享的资源。

    单例模式的实现

    实现单例模式的几个要素：

  • 构造方法是私有的

  • 私有静态引用指向自己的实例

  • 公有的静态方法返回自己的实例

饿汉式（推荐）

/**
 * Created on 2020/6/3
 *
 * @author Tu ShengTao
 * @Description: 单例模式之饿汉形式
 * 饿汉形式：在该类被加载进jvm的时候就创建该实例,只会被加载一次
 *
 */
public class HungrySingleton {

    private static final HungrySingleton singleton = new HungrySingleton();

    private HungrySingleton(){};

    public static HungrySingleton getInstance(){
        return singleton;
    }

    /**
     * 主方法测试
     */
    public static void main(String[] args){
        // 模拟在其他类进行调用
        // 由于HungrySingleton的构造方法私有，所以无法实例化（即无             法new出一个对象）
        // 使用HungrySingleton的静态方法getInstance()获取唯一实例

        // 未重写Object的hashcode()时默认返回实例/对象的哈希码
        // 未重写的hashcode()使用实例/对象的内存地址生成哈希码
        // 此处指的哈希码：是堆内存地址的映像,通过某种规则可以使用哈希            码计算得到对象的堆内存地址
        // 拓展知识提示：HashMap、hashcode()、equals()、哈希表
        System.out.println(HungrySingleton.getInstance().hashCode());
    }
}

懒汉式（不推荐）

想象一下这个多线程的场景，多个线程同时工作意味着有可能创建了多个实例，因为没有给getInstance()加锁。

/**
 * @author Tu ShengTao
 * Description: 单例模式之懒汉形式，不推荐
 */
public class LazySingleton {
    private static LazySingleton lazySingleton ;
    private LazySingleton(){}
    public static LazySingleton getInstance(){
        if (lazySingleton == null){
            // 用于多线程测试 可删
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            // 核心代码
            lazySingleton = new LazySingleton();
        }
        return lazySingleton;
    }

    /**
     * 主方法测试
     */
    public static void main(String[] args) {
        // 可能会输出多个不同的哈希码，说明生成了多个实例
        for (int i = 0; i < 100; i++) {
            new Thread(()->{
           System.out.println(LazySingleton.getInstance().hashCode());
            }).start();
        }
    }
}

懒汉式优化

/**
 *
 * @author Tu ShengTao
 * Description: 单例模式之懒汉形式 优化 ：无论开多少个线程只有一个实例
 */
public class LazySingleton2 {
    private static LazySingleton2 lazySingleton2 = null ;
    private LazySingleton2(){}
    public static synchronized LazySingleton2 getInstance(){
        if (lazySingleton2 == null){
            try {
                Thread.sleep(400);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            lazySingleton2 = new LazySingleton2();
        }
        return lazySingleton2;
    }
    /**
     * 主方法测试
     */
    public static void main(String[] args) {
        for (int i = 0; i < 100; i++) {
            new Thread(()->{
                System.out.println(LazySingleton2.getInstance().hashCode());
            }).start();
        }
    }
}

单例模式总结

什么时候用？
答: 在一个系统或者模块只需要一个类的实例的时候
怎么实现单例？
答：1类的构造方法私有、2在本类中new一个实例 instance、3在本类创建一个 public、 static 成员方法且返回本类的一个实例 instance

策略模式

策略模式(Strategy Pattern)：定义一系列算法，将每一个算法封装起来，并让它们可以相互替换，策略模式属于行为模式。策略模式能够达到开闭原则(软件设计原则的一种)的目标：对修改关闭，对拓展开放；通俗点解释开闭原则：当有新的业务需求时，就是不修改原来的代码，在原来代码的基础上进行拓展。下面将结合实际案例来进行理解。

使用场景

  • 如果在一个系统里面有许多类，它们之间的区别仅在于它们的行为，那么使用策略模式可以动态地让一个对象在许多行为中选择一种行为。
  • 一个系统需要动态地在几种算法中选择一种。
  • 如果一个对象有很多的行为，如果不用恰当的模式，这些行为就只好使用多重的条件选择语句来实现。
  • 不希望客户端知道复杂的、与算法相关的数据结构，在具体策略类中封装算法和相关的数据结构，提高算法的保密性与安全性。

    应用案例

    • 诸葛亮的锦囊妙计，每一个锦囊就是一个策略。
    • 旅行的出游方式，选择骑自行车、坐汽车，每一种旅行方式都是一个策略。
    • 做鱼的方式，可以选择不同的做法，如清蒸、红烧，每一种做法就是一种策略。

策略模式结构与实现

策略模式的结构


策略模式的主要角色如下：

  1. 抽象策略（Strategy）类：定义了一个公共接口，各种不同的算法以不同的方式实现这个接口，环境角色使用这个接口调用不同的算法，一般使用接口或抽象类实现。

  2. 具体策略（Concrete Strategy）类：实现了抽象策略定义的接口，提供具体的算法实现。

  3. 环境（Context）类：持有一个策略类的引用，最终给客户端调用。

策略模式的实现

现在我们构造一个场景，以做鱼为案例：儿子告诉妈妈说想吃鱼，妈妈问儿子想要哪种吃法（吃法有很多：红烧、清蒸、烤），儿子回答说想吃红烧鱼，之后妈妈就去准备了。试问：如果用面向对象如何实现？是不是想到用if-else了？如果儿子回答是红烧鱼，那妈妈就做红烧鱼；如果儿子回答是清蒸鱼，妈妈就做清蒸鱼；如果儿子回答是烤鱼，妈妈就做烤鱼。那如果用if-else的话就违背上面提到的开闭原则，如果儿子突然有一天想吃炸鱼，那又得加一个判断语句，这样程序的可维护性不是很好。如何不用条件判断语句呢？下面我们来揭晓答案！

策略模式案例角色说明

  1. 抽象策略类——CookingFish.java：定义了一个cooking()方法，各种不同的做法以不同的方式实现这个方法，环境角色(Mother.java)使用这个接口调用不同的做法。

  2. 具体策略类——BrisedFish.java和SteamedFish.java：实现了抽象策略定义的接口，提供具体的做法实现，BrisedFish意思是红烧鱼，SteamedFish意思是清蒸鱼。

  3. 环境类——Mother.java：持有一个策略类的引用，最终给客户端(Son.java)调用，在这个案例中儿子充当客户端。

抽象策略类：

/**
 * 鱼烹饪

环境类——妈妈类

/**
 * 妈妈。
 *
 * @author tushengtao
 * @author: tushengtao
 * @Description: 环境类 暴露给客户端使用的类
 * 举例：你想吃红烧鱼了，你不用管红烧鱼具体的做法，你只要告诉你妈妈你要吃红烧鱼，你妈妈就会去给你做了
 * @date 2020-08-22 2:03
 */
public class Mother {
    private CookingFish cookingFishStrategy;

    public CookingFish getCookingFish() {
        return cookingFishStrategy;
    }

    public void setCookingFish(CookingFish cookingFishStrategy) {
        this.cookingFishStrategy = cookingFishStrategy;
    }
    public void cookingFish(){
        System.out.println("好，儿子！你想吃什么鱼我就给你做什么鱼!等着......");
        cookingFishStrategy.cooking();
    }
}

/**

策略模式总结

什么时候用？
答: 系统未来可能需要更换或者增加一些算法，需要动态地在几种算法中选择一种；你想避免使用大量的条件选择语句；希望在具体策略类中封装算法和与相关的数据结构。总而言之就是你想尽可能遵循开闭原则的时候。
怎么实现策略模式？
答：1创建抽象策略类 、2创建不同的具体策略类且要实现抽象策略类、3创建环境类，要创建一个对抽象策略类的引用实例 、4 客户端类调用环境类

代理模式

代理模式(Proxy Pattern) ：给某一个对象提供一个代 理，并由代理对象控制对原对象的引用，通俗讲就是一个类代表另一个类的功能，比如中介代表的是房东，房东全权委托给中介，中介可以直接和租客沟通租房事务，而房东无需过问细节。代理模式的英 文叫做Proxy或Surrogate，代理模式是一种结构型模式。

使用场景

  • 远程代理，这种方式通常是为了隐藏目标对象存在于不同地址空间的事实，方便客户端访问。例如，用户申请某些网盘空间时，会在用户的文件系统中建立一个虚拟的硬盘，用户访问虚拟硬盘时实际访问的是网盘空间。
  • 虚拟代理，这种方式通常用于要创建的目标对象开销很大时。例如，下载一幅很大的图像需要很长时间，因某种计算比较复杂而短时间无法完成，这时可以先用小比例的虚拟代理替换真实的对象，消除用户对服务器慢的感觉。
  • 安全代理，这种方式通常用于控制不同种类客户对真实对象的访问权限。
  • 智能指引，主要用于调用目标对象时，代理附加一些额外的处理功能。例如，增加计算真实对象的引用次数的功能，这样当该对象没有被引用时，就可以自动释放它。
  • 延迟加载，指为了提高系统的性能，延迟对目标的加载。例如，Hibernate 中就存在属性的延迟加载和关联表的延时加载。

    应用案例

1、Windows 里面的快捷方式。 2、猪八戒去找高翠兰结果是孙悟空变的，可以这样理解：把高翠兰的外貌抽象出来，高翠兰本人和孙悟空都实现了这个接口，猪八戒访问高翠兰的时候看不出来这个是孙悟空，所以说孙悟空是高翠兰代理类。 3、买火车票不一定在火车站买，也可以去代售点。 4、一张支票或银行存单是账户中资金的代理。支票在市场交易中用来代替现金，并提供对签发人账号上资金的控制。 5、Spring Aop。

代理模式结构与实现

代理模式结构

代理模式的主要角色如下：

  1. 抽象角色——RentHouse.java：定义代理角色和真实角色的公共对外方法，一般使用接口或者抽象类。
  2. 真实角色——Landlord.java：定义真实角色要实现的业务，提供给代理角色调用。
  3. 代理角色——Proxy.java：代理真实角色，通过真实角色的业务来实现抽象方法，一般会有附属业务。
  4. 客户——Client.java：访问代理角色。

代理模式实现

静态代理实现

我们根据房东租房这个案例来进行实现。房东想租房，但自己又懒，这个时候就需要一个中介了，中介帮助房东处理租房的事情，这样的话，租房者就直接和中介进行沟通，租房者不需要再和房东沟通。中介就代理了房东，在租房前后再附加其他业务操作，如：租房者看房，签合同……

代理模式案例角色说明
  1. 抽象角色：租房业务
  2. 真实角色：房东
  3. 代理角色：中介，可能还有带客户看房子等业务
  4. 客户：租房者

抽象角色——租房业务类

/**
 * 抽象角色 租房业务
 *
 * @author tushengtao
 * @date 2020/08/21
 */
public interface RentHouse {
    public void rent();
}

真实角色——房东类

/**
 * @author: tushengtao
 * @Description: 真实角色 房东
 * @date 2020-08-21
 */
public class Landlord implements RentHouse{
    @Override
    public void rent() {
        System.out.println("房东出租房子");
    }
}

代理角色——中介类

/**
 * @author: tushengtao
 * @Description: 代理角色 中介
 * @date 2020-08-21
 */
public class Proxy implements RentHouse {

    private Landlord landlord;
    public Proxy(Landlord landlord){
        this.landlord = landlord;
    }
    /**
     * 代理租房子
     */
    @Override
    public void rent() {
        checking();
        landlord.rent();
        signContract();
        collectFees();
    }
    /**
     * 看房
     */
    public void checking() {
        System.out.println("中介带你看房");
    }
    /**
     * 签署合同
     */
    public void signContract() {
        System.out.println("签租赁合同");
    }
    /**
     * 收中介费
     */
    public void collectFees() {
        System.out.println("收中介费");
    }
}

客户端类——租房者类

/**
 * @author: tushengtao
 * @Description: 客户端类 租房者
 * @date 2020-08-21
 */
public class Client {
    public static void main(String[] args) {
        // 房东租房子
        Landlord landlord  = new Landlord();
        // 代理，中介帮忙租房子，并且有其他附属操作
        Proxy proxy = new Proxy(landlord);
        // 不需要找房东,直接找中介租
        proxy.rent();
    }
}

代理模式总结

什么时候用？
答: 1当我们想要隐藏某个类时，可以为其提供代理。2当一个类需要对不同的调用者提供不同的调用权限时，可以使用代理类来实现（代理类不一定只有一个，我们可以建立多个代理类来实现，也可以在一个代理类中进行权限判断来进行不同权限的功能调用）。3当我们要扩展某个类的某个功能时，可以使用代理模式，在代理类中进行简单扩展（只针对简单扩展，可在引用委托类的语句之前与之后进行）。
怎么实现静态代理？
答：静态代理 ： 1创建抽象业务类 、2创建真实角色类，实现抽象接口的方法 、3创建代理角色类，实现抽象接口的方法，并且创建一个真实角色类引用，拓展其他业务方法、4 客户端类实例化一个真实角色类，实例化一个代理角色类并且将真实角色实例作为参数传入代理类构造方法。
